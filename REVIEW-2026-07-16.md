# Ledger Budget App — Full UX + Finance Review (2026-07-16)

Reviewed the local source (Windows Read tool = source of truth), the LIVE deployed files on Vercel,
and ran a real React render of the app (jsdom, mobile viewport) tracing the salary → 401k gear →
dashboard math end-to-end. Changes are batched, prioritized by return on time, and sized to paste
directly into a Claude session one batch at a time (each is well under the truncation threshold).

**Verification results first:** the finance math is correct. Take-home model, month-gated pretax
deductions (salary stopping in June halves the annual deduction), gross = take-home + deductions,
and the savings KPI reconciling exactly with the balance roll-forward at 0% yield — all asserted
programmatically, all passed, zero JS errors. The problems are not in the math; they are in a
corrupted stylesheet and a few trust/layering issues.

## THE headline finding

**The deployed styles.css on Vercel is truncated.** Near the end of the mobile media block it stops
mid-declaration:

```
  .btn {
    min
```

…and the auth-screen CSS was appended after the break. Consequences on the live site:

1. Everything from that point in the mobile block is dead: 44px button/input touch targets, KPI
   font sizing for half-width cards (numbers overflow), allocation-card padding, foot-note
   stacking, the entire mobile transaction-card layout (transactions render as a cramped
   5-column table), milestone spacing, and the enlarged ⓘ tap targets.
2. The `@media (max-width: 768px)` block is never closed, so **all auth-screen styles are
   swallowed** by the malformed rule — a CSS parser check confirmed zero `.auth-*` rules survive.
3. Depending on the browser's error recovery, some engines drop even more of the mobile block.

This is almost certainly the root of "cramped, out of line" on mobile. The intact tail exists in
`styles.2026-07-14.backup.css` (lines 1152–1202), so the repair is a pure splice — Batch 1 below.

How it happened: a truncated file write (the known stale-mount issue) got committed and deployed.
The new skill (below) makes "verify deployed file integrity" the first step of every review.

## Batch 2 — Make autosave trustworthy (remove the fake Save button, show sync status) (P1)

The "Save budget" button is a placebo — `handleSave` only flips a "Saved ✓" label. Data actually
auto-saves (localStorage instantly + Supabase on an 800ms debounce). Two real gaps: closing the tab
within the debounce window can lose the last edit, and cloud-save errors only go to the console —
a user could edit for a month on a broken RLS policy and lose everything silently.

Paste to Claude:

> In the Ledger Budget App, make autosave the visible, trustworthy save path:
>
> 1. `js/budget.jsx` (BudgetScreen): delete the `saved` state line, the `handleSave` function, and
>    the `<button className="btn primary" onClick={handleSave}>…Save budget…</button>` in the
>    page-head actions. The foot-note already says "Changes saved automatically."
> 2. `js/app.jsx` (App): add `const [syncState, setSyncState] = React.useState('idle');`
>    In the debounced cloud-save effect, call `setSyncState('saving')` when scheduling, and in the
>    upsert `.then`, `setSyncState(error ? 'error' : 'saved')`.
> 3. Same file: add a flush-on-exit effect so the last edit isn't lost to the debounce:
>    ```jsx
>    React.useEffect(() => {
>      const flush = () => {
>        if (!window.sb || !userId || !saveTimer.current) return;
>        clearTimeout(saveTimer.current); saveTimer.current = null;
>        window.sb.from('ledger_state')
>          .upsert({ user_id: userId, data, updated_at: new Date().toISOString() })
>          .then(({ error }) => setSyncState(error ? 'error' : 'saved'));
>      };
>      const onVis = () => { if (document.visibilityState === 'hidden') flush(); };
>      document.addEventListener('visibilitychange', onVis);
>      window.addEventListener('pagehide', flush);
>      return () => { document.removeEventListener('visibilitychange', onVis);
>                     window.removeEventListener('pagehide', flush); };
>    }, [data, userId]);
>    ```
> 4. Topbar (next to the avatar), render a status pill when signed in:
>    saving → "Saving…", saved → "Saved ✓", error → "Not saved — tap to retry" (onClick re-runs the
>    upsert). Hide when idle. Style `.sync-pill` small/muted; `.sync-pill.error` uses `var(--warn)`
>    and is a real ≥44px-tall touch target on mobile.
> 5. Shared-device privacy: in `signOut`, clear `ledger.data.v2`, `ledger.ui.screen`, and
>    `ledger.ui.month` from localStorage. Otherwise the next person who creates an account on the
>    same browser seeds their new cloud budget with the previous user's data (first-sign-in seeding
>    in the cloud-load effect).
>
> Verify: edit a cell, see "Saving…" then "Saved ✓"; kill network in DevTools, edit, see the error

## Batch 3 — Fix "popups fall behind text" + Escape to close (P1)

Root cause found: the ⓘ popover (`.info-pop`) is `position:fixed` with `z-index:300`, but it's
rendered INSIDE a sticky sheet cell that has `z-index:3`. A sticky element with a z-index creates a
stacking context, so the popover's 300 only counts within that cell — sticky cells of LATER rows
(also z-index 3, later in paint order) draw on top of it. That's exactly "popup behind other text,
unreadable." Portaling the popover to `document.body` takes it out of every stacking context.

Paste to Claude:

> In the Ledger Budget App:
>
> 1. `js/shared.jsx`, `Info` component: render the popover through a portal. Replace
>    `{pos && <span className="info-pop" style={{ position:'fixed', left:pos.x, top:pos.y, width:pos.w }}>{text}</span>}`
>    with
>    `{pos && ReactDOM.createPortal(<span className="info-pop" style={{ position:'fixed', left:pos.x, top:pos.y, width:pos.w }}>{text}</span>, document.body)}`
>    (Reason: the ⓘ sits inside sticky sheet cells whose z-index traps the fixed popover in a
>    stacking context; later sticky cells paint over it.)
> 2. `js/shared.jsx`: add a tiny global hook —
>    ```jsx
>    function useEscapeClose(onClose) {
>      React.useEffect(() => {
>        const onKey = (e) => { if (e.key === 'Escape') onClose(); };
>        document.addEventListener('keydown', onKey);
>        return () => document.removeEventListener('keydown', onKey);
>      }, [onClose]);
>    }
>    ```
>    Call `useEscapeClose(onClose)` at the top of RowSettingsModal, DebtRateModal, GivingGoalModal
>    (budget.jsx), LogExpenseModal (track.jsx), AddMilestoneModal (plan.jsx), and use it for the
>    Data menu in app.jsx (`useEscapeClose(() => setShowDataMenu(false))` — but only while open, so
>    guard: put the hook inside a small wrapper or condition the handler on `showDataMenu`).
> 3. Add `role="dialog" aria-modal="true"` to each `.modal` div and `aria-label`s where missing.
>
> Verify: open the ⓘ on "Beginning Cash Savings" and scroll the sheet — the popover must sit above
> every row; press Escape in each modal.

## Other considerations (no batch, but know these)

- **In-browser Babel is your biggest performance tax.** ~120KB of JSX is compiled on the phone on
  every load, plus React/Babel/Supabase come from three different CDNs (unpkg + jsdelivr). If any
  CDN hiccups, the app is down. Fine for now; when you feel "slow first load" pain, the fix is a
  small build step (Vite) — a bigger session, plan it separately.
- **Multi-device is last-write-wins.** Two devices open at once will silently overwrite each other
  (no updated_at conflict check). Acceptable for single users; worth a warning if couples share an
  account on two phones.
- **Starter data ships a 'Retirement (401k)' category** — the exact double-count trap the gear
  modal warns about. Consider removing that row from STARTER_DATA (keep Roth IRA + Cash Savings)
  so new users never step in it.
- **Cloud-load race:** edits made in the first second after sign-in (before the cloud row loads)
  are overwritten by the cloud copy. Rare; fix only if someone reports it.
- **Repo hygiene is fine** — backups are gitignored. `ledger-ux-review.html` and old `index.*.backup`
  files still in the repo can be deleted via GitHub Desktop whenever.
- **Process:** the truncation got committed because verification happened before the write, not
  after at the file tail. The replacement app-reviewer skill now makes deployed-file integrity
  check step 1 of every review.
