# Ledger Budget App — Financial & UX Review (2026-07-09)

Reviewed against the live deployed code (commit `27c0067`, index.html 1,766 lines + data.js + styles.css).
Changes are listed by priority: **5 = fix first / highest return**, 1 = polish.

## Priority 5 — Core calculation correctness

### 5.1 Fix the pretax income flow (the reported gear-icon bug)
**Root cause:** the app treats the income rows in the sheet as TAKE-HOME pay, and payroll deductions entered in the gear modal are added ON TOP to derive gross (`grossIncomeAnnual = incomeAnnual + deductionMonthlyTotal * 12`, index.html ~line 107). So entering $500 for 401(k) changes nothing visible except a new "Gross income" KPI — "Annual income" and "Total receipts" stay the same, which reads as broken.

Implement all of the following:
1. When any payroll deduction is set (`payrollActive`), rename the KPI "Annual income" → "Take-home income" so the two KPIs read "Take-home income" / "Gross income".
2. In the sheet's Receipts section, when `payrollActive`, render two read-only computed rows under the income rows: "Pretax payroll deductions" (monthly = sum of all rows' deductions, only counted in months where that income row's take-home > 0) and "Gross receipts" = Total receipts + deductions. This makes the $500 visibly flow through.
3. Fix annualization: `deductionMonthlyTotal * 12` assumes 12 active months. Compute per row: deductions count only in months where `row.monthly[m] > 0`. Apply the same month-gating to `retHsaMonthly` inside `ltSavingsMonthly` (currently adds 401k+HSA to the LT balance all 12 months even if income stops in June) and to `retHsaAnnual`.
4. Double-count guard: on saving income settings with `retirement401k > 0` or `hsa > 0`, scan budget categories for names matching /401k|403b|retirement|hsa/i; if found, show a confirm: "You also budget $X/mo to '<category>'. If that's the same money, it will be counted twice — keep the payroll deduction and delete the category (recommended), or cancel." Note: the starter data ships with a 'Retirement (401k)' category, actively creating this trap.

### 5.2 Reconcile the "Annual savings" KPI with the balance rows
`effectiveSaveAnnual = bucketAnnual.save + netAnnual + retHsaAnnual` (KPI includes unallocated net income), but the monthly balance accumulation (`cashSavingsMonthly`) only accumulates save-bucket categories — unallocated net never lands in any balance. So "Annual savings" says one number while "Total savings (Cash + LT)" grows by a different one.
**Fix:** add `netMonthly[m]` to `cashSavingsMonthly[m]` in the balance roll-forward (matches the stated design that under-allocated income flows to savings; over-allocation correctly drags it down). Verify: with 0% yields, endCash[11]+endLT[11] − beginning balances must equal effectiveSaveAnnual exactly.

### 5.3 Mobile: settings gear and delete buttons are invisible on touch devices
`.row-gear` and `.row-del` are `opacity: 0` and only revealed on row `:hover` (styles.css ~line 380–405). Phones have no hover, so the ONLY entry point to fill mode, Fund classification, and payroll deductions is undiscoverable on mobile, and targets are 20px (spec minimum 44px).
**Fix:** inside the `@media (max-width: 720px)` block set `.row-gear, .row-del { opacity: 0.55; }` and give them a ≥44×44px hit area (padding, negative margin to preserve layout). Also enlarge the `.bucket-dot` tap area to ≥32px via a transparent padded hit box.

## Priority 4 — Consistency and mobile blockers

### 4.1 "Year-end total wealth" KPI ignores debt
Budget tab KPI = `endCash[11] + endLT[11]` only, while the Plan tab's "Projected net worth" = cash + LT − debt. If Beginning Debt > 0 the two tabs disagree.
**Fix:** subtract `endDebt[11]` and, when `beginningDebt > 0`, label it "Year-end net position"; keep "Year-end total wealth" otherwise.

### 4.2 Allocation bar mixes take-home and gross baselines
`AllocationSummary` uses take-home income as denominator for the bar/legend, but when `payrollActive` the "Savings rate" rule uses gross — two different percentage baselines shown side by side, and the header says "every dollar earned" while pretax dollars are missing from the bar.
**Fix:** when `payrollActive`: use gross as the denominator everywhere in this component, add a "Pretax savings" segment (401k+HSA, save color) and a muted "Other payroll deductions" segment (premiums + other pretax), and show gross in the header income line. When not payrollActive, leave exactly as is.

### 4.3 Modals can overflow small screens with no scroll
`.modal` has fixed width (440px income settings, tall with 5 number fields + reconciliation box) and no max-height.
**Fix:** add to `.modal`: `max-height: calc(100dvh - 32px); overflow-y: auto;` and give `.modal-overlay` `padding: 16px` + safe-area inset.

### 4.4 Plan chart is sticky on mobile and overlaps the sliders
The chart card has inline `position:'sticky', top:80` (PlanScreen JSX) plus a CSS sticky rule. On mobile the plan-grid collapses to one column, so the sticky chart floats over the slider controls as you scroll.
**Fix:** remove the inline style; keep stickiness in CSS only (`.plan-grid > .chart-card { position: sticky; top: 80px; }`) and set `position: static` for it inside the 720px media block.

## Priority 3 — Accuracy edge cases and spec alignment

### 3.1 Name-regex classification is fragile (real money-routing risk)
Cash vs long-term savings uses `/long.?term|retirement/i` and debt payments use `/debt|interest/i` on the category NAME. "Roth IRA", "Brokerage", "Index funds" silently route to CASH; "Student loan", "Car payment" are NOT treated as debt payments, so Beginning Debt never declines.
**Fix:** add a "Counts as" selector in the category gear modal: Auto (name-based) / Cash savings / Long-term investing / Debt payment. Store as `c.flow`; helpers check the override first, regex as fallback. Broaden the debt regex to include /loan|payment.*(car|auto|student)|mortgage payoff/i is NOT recommended — the explicit override is cleaner.

### 3.2 Track: expenses logged to another month/year silently disappear
`monthTxs` filters to `data.year` + viewed month; the Log Expense date field is unconstrained. Logging with a date in a different month saves the transaction but it vanishes from the current view.
**Fix:** after save, if the transaction's month ≠ viewed month, switch `monthIdx` (and warn if the year ≠ `data.year`: "This expense is dated 2025 — it won't appear in your 2026 views.").

### 3.3 Mobile spec alignment
- Change the breakpoint from 720px to 768px (design spec: mobile <768).
- Audit touch targets to ≥44px: month chips (currently ~34px tall), sheet number inputs' row height, year-pill arrows, modal close buttons.
- Ensure the FAB (bottom: 76px) doesn't cover the "Save expense" / "Save budget" buttons on short screens.

### 3.4 Milestone math inconsistency
"% by yr N" uses `finalBal` (cash+LT−debt) but the ETA simulation (`monthsToReachTotal`) ignores debt entirely.
**Fix:** use cash+LT for both (milestones are asset targets; the separate Debt-free milestone already handles debt).

## Priority 2 — Data lifecycle and guidance

### 2.1 CSV import wipes gear settings and can't round-trip
Import replaces all income rows and groups with fresh objects — payroll deductions, Fund flags, and fill modes are lost, with no warning. The CSV format has no columns for them.
**Fix:** (a) add to the import confirm dialog: "Row settings (payroll deductions, Fund, fill mode) will be reset."; (b) preserve settings for rows whose Name matches an existing row (case-insensitive); (c) add an "Export current budget (CSV)" button next to "CSV template".

### 2.2 Year navigation is misleading
The year pill changes `data.year`, which relabels the budget and filters Track transactions — but there is only ONE budget shared across all years.
**Fix (minimal):** first time the year is changed, show a one-time notice: "Ledger keeps one budget. Changing the year relabels it and shows only that year's logged expenses." A true per-year budget archive is a larger future feature.

### 2.3 Move instruction text into contextual tooltips
Per the design constraints: keep the collapsible Instructions block but retitle "Key notes", and add small ⓘ tooltips at point of use — on the Beginning-balance rows, the bucket legend, the gear icon (first-run pulse animation on mobile so it's discoverable), and the Fund badge.

## Other considerations (process & pitfalls)

- **index.html was missing from your local folder** — only dated backups existed (the known filesystem-sync issue). I restored it from the deployed git commit `27c0067`; md5 verified identical, script tags balanced (8/8), tail intact. GitHub Desktop should show no pending change for it.
- **Take-home vs gross is a product decision, not just a bug.** Fix 5.1 keeps the current "sheet = take-home" model and makes it visible. The alternative (users enter gross and the app derives take-home) is more intuitive for W-2 users but a bigger rework. Decide before implementing 5.1.
- **index.html is 1,766 lines / ~97 KB — over the 800-line truncation threshold.** Per the github-vercel-app-coding procedure: back up before editing, apply changes with asserting scripts (not full-file rewrites), and verify script-tag balance + tail after every edit. Consider splitting the four inline `<script type="text/babel">` blocks into separate .jsx files served the same way (the README already describes that layout) to shrink future edit risk.
- **All data is localStorage-only.** For family/friends this is fragile: clearing the browser or switching devices loses everything, and Safari evicts localStorage after 7 days of inactivity on some setups. Near-term: an in-app "last backup X days ago — export now" reminder. Longer-term: Supabase persistence like your other apps.
- **Verify live after deploying:** after pushing these changes, load the Vercel URL on a phone, set a $500 401k deduction, and confirm: Gross = take-home + $6,000, Annual savings rises by $6,000, and year-end wealth rises by ~$6,000+yield. That single trace exercises fixes 5.1–5.2.
